## 三、RICE评分模型

### 评分标准
| 维度 | 评分标准 | 评分依据说明 |
|------|---------|------------|
| **R**each(影响范围) | 1分:<10%用户 / 3分:10-50% / 5分:>50% | 基于用户数据或合理估算，标注数据来源 |
| **I**mpact(影响程度) | 1分:微小 / 2分:低 / 3分:中 / 5分:高 | 对北极星指标的预期贡献程度 |
| **C**onfidence(信心度) | 0.5:低(直觉) / 0.8:中(定性数据) / 1.0:高(定量验证) | 标注信心来源：直觉/访谈/数据 |
| **E**ffort(工作量) | 1:<1周 / 2:1-2周 / 3:2-4周 / 5:>4周 | 含设计+开发+测试全链路工时 |

### RICE计算公式
> **总分 = (Reach x Impact x Confidence) / Effort**
>
> 示例：功能A → R:5 × I:3 × C:0.8 / E:2 = **6.0分**

| 2 | | | | | | | P0 | 纳入MVP |
| ... | | | | | | | | |

## 四、KANO模型分析

| 需求类型 | 定义 | 识别方法 | 本产品对应功能 | 策略 |
|---------|------|---------|-------------|------|
| **基本型(Must-be)** | 没有会不满，有了不加分 | 用户认为理所当然的功能 | | 必须100%满足，做到行业基准线 |
| **期望型(One-dimensional)** | 做得越好满意度越高 | 用户能明确表达的需求 | | 重点投入，争取超预期 |
| **兴奋型(Attractive)** | 没有不会不满，有了会惊喜 | 用户自己都没想到的体验 | | 选择性投入，作为差异化亮点 |
| **无差异型(Indifferent)** | 有没有都不影响满意度 | | | 降低投入或砍掉 |
| **反向型(Reverse)** | 有了反而不满 | | | 明确不做 |

# Few-shot Example

> **功能清单示例（简版）——AI阅读助手App**
>
> | 序号 | 功能点 | 用户故事 | KANO | RICE | 优先级 | 验收标准 |
> |------|--------|---------|------|------|--------|---------|
> | 1 | AI摘要生成 | 作为忙碌的职场人，我想一键获取文章核心摘要，以便在3分钟内了解万字长文要点 | 期望型 | R5×I5×C1.0/E3=8.3 | P0 | Given 用户上传文章，When 点击"生成摘要"，Then 30s内返回≤300字摘要，准确率≥85%(人工评测) |
> | 2 | 阅读进度同步 | 作为多设备用户，我想在手机和电脑间同步阅读进度，以便无缝切换阅读 | 基本型 | R3×I2×C1.0/E2=3.0 | P1 | Given 用户在设备A阅读至50%，When 在设备B打开同一文章，Then 自动定位至50%位置，延迟≤3s |
>
> **KANO决策**：AI摘要(期望型)+P0 → MVP核心卖点；阅读进度同步(基本型)+P1 → V1.0必须满足

# Constraints
- 每个功能须有完整的用户故事(As a...I want...So that...)
- 每个功能须有量化验收标准(Given-When-Then格式)
- RICE评分须逐维度打分，标注计算过程和评分依据
- 须为每个功能标注KANO分类
- P0功能须控制在合理范围（MVP阶段控制在5-7个）
- 优先级须附理由，说明为什么是P0/P1/P2
- 须标注功能间的依赖关系，并用Mermaid图可视化
- 砍掉的功能须记录原因和重新评估条件

# Temperature Guidance
- 建议 Temperature: 0.2-0.3
- 功能清单要求高度结构化和精确性，RICE评分须一致性计算；功能描述和用户故事部分可适当提高至0.4
